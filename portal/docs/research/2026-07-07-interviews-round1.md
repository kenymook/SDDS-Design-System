# Интервью с пользователями — раунд 1 (финальный отчёт)

**Дата сессий:** 07.07.2026 · **Отчёт собран:** 09.07.2026
**Прототип:** https://portal-flow-prototype.vercel.app

Три интервью с живыми пользователями по SDDS Portal и DS Builder. Это сводный отчёт: индивидуальные разборы по каждому респонденту лежат отдельно (ссылки в конце), а здесь — что повторяется у разных ролей, где согласие, приоритеты и конкретный план работ. Отчёт объединяет два независимых разбора одних и тех же интервью — они сошлись почти во всём, что само по себе повышает доверие к находкам.

## Executive summary

**Что подтвердилось:**
- SDDS как дизайн-система считывается быстро — все трое своими словами описали «компоненты + темы брендов» и механизм токенов.
- Live demo хорошо объясняет смену бренда/темы (white-label без кода).
- Builder в базовом сценарии проходим: люди зашли, поменяли цвет, опубликовали.
- Идея Version/CLI и модель «обновление как в Git» вызывает **доверие**, особенно у разработчиков.
- Документация воспринимается как важная часть продукта, не вторичная справка.

**Что не сработало:**
- Первый экран лендинга не объясняет, что это ещё и **инструмент/handoff**, а не только готовая ДС.
- Разработчикам не хватает практической документации: установка, фреймворки, props, код, CLI.
- **Publish непонятен**: люди не понимают, это «сохранение», «раскатка в прод», «Git», «CI/CD» или «конфигурация для разработки».
- В Builder иконки сайдбара без подписей читаются плохо — панель игнорируют.
- Редактор цвета не совпадает с ментальной моделью дизайнера: он ждёт общую палитру/бренд-цвет, а попадает в семантические токены.

Одной фразой: **портал уже доказывает «SDDS существует и работает», но пока не доказывает «на этом реально можно работать командой».**

## Что работает (консенсус)

- **Назначение продукта считывается быстро** — у всех троих.
- **Live demo доказывает идею** смены бренда без кода.
- **Builder проходим для базовых действий** — создание проекта, правка цвета, публикация выполнялись самостоятельно.
- **Git-аналогия обновления и handoff внушает доверие** — «получение изменений через Version/CLI» воспринимается естественно (явно — backend; дизайнер назвал единый источник правды токенов **самым полезным**: «это коннектится в Figma и раскатывается в разработку — основной смысл»).
- **Экран Health заинтересовал** дизайнера («показывать, сколько компонентов покрыто токенами, а где захардкожено — полезно, если докрутить»).

## Приоритеты

Приоритет = серьёзность × охват (сколько респондентов независимо отметили). «Кто» — **F**rontend, **B**ackend, **D**esigner.

### 🔴 P0 — критично

**1. Лендинг не продаёт главный смысл** · *D (эмоц.), F/B частично*
Первый экран читается как «готовые компоненты, токены, бренды от Сбера». Ценность инструмента (Builder + handoff) люди понимают только внутри. Дизайнер оценил понятность с лендинга на **3/10** и сравнил продукт с shadcn — «не понял, что это доп. инструмент».
→ На первом экране сказать: **SDDS — дизайн-система и инструмент, который связывает дизайн и разработку через токены, версии и CLI.**

**2. Нет developer quick start** · *F B D*
Разработка спросит (и это ждут все трое): как установить; поддерживаются ли React/Vue/другие; где props; где код; где GitHub; как использовать опубликованную Version. Без этого портал убедителен для дизайнера, но не готов «к продаже команде».
→ Раздел для разработчиков (состав — в плане, шаг 2). **Упирается в открытые решения:** контракт CLI и перечень изменяемых свойств компонентов ещё не закрыты — целевую модель показываем, детали честно помечаем «в проработке».

**3. Publish непрозрачен** · *F B D*
Ключевой вопрос дизайнера: «Оно улетело в Git разработчикам или прокатилось на прод?». Что происходит после Publish — не понял никто.
→ Явно объяснить, что Publish (сверено с [моделью MVP](../deliverables/02-ds-builder-mvp-scope.md)): создаёт **immutable Version**; сохраняет **Published Configuration**; **не** раскатывает в прод; **не** генерирует target-файлы (это забота CLI); отправляет уведомления участникам; становится доступен разработчику через CLI *(контракт CLI — в проработке, так и помечаем)*.

**4. Навигация Builder неочевидна** · *F B D*
Иконки в узком сайдбаре игнорируются или не считываются — все трое.
→ Минимум: tooltip на hover/focus, подпись активного раздела, понятные иконки для Colors / Sizes / Fonts / Components / Health / Versions / Publish. Лучше — expand-on-hover rail (компактный режим как в IDE).

**5. Changes недостаточно наглядны** · *F (🔴), D*
Разработчик коротко: «Покажите цвет до и после». Вместо списка токенов ждут визуальный diff.
→ Swatch/значение **до → после**, preview компонента before/after, список затронутого, кнопка «к источнику».

## Разрез по ролям

**Разработчики (F + B) — про внедрение.** Главный запрос — практика: как подключить, что поддерживается, код и props, полный CLI, changelog в консоли после обновления. Frontend болезненнее всего отреагировал на навигацию браузера (Back/новая вкладка). Backend сильнее давит на онбординг, «ведение» по сценарию (ссылки из доки → в разделы Builder) и персонализацию (релизы только свои, прогресс за авторизацией).

**Дизайнер (D) — про ментальную модель редактора и позиционирование.** Дважды не связал «меняю бренд-цвет» с тем, что правит `text&icon primary` — ждёт общую палитру с каскадом. Не понял из лендинга, что это инструмент. При этом именно он яснее всех сформулировал ценность продукта и увидел её **только пройдя Builder руками**, не с лендинга.

## План работ

Порядок — по «максимум ценности при минимуме риска». Большинство пунктов — **уточнения к существующему MVP**, а не новый scope; отдельно помечено, что упирается в незакрытые решения.

**Шаг 1. Пересобрать первый экран Portal.** Донести «ДС + инструмент + handoff» выше линии сгиба: «Настраивайте темы в Builder», «Публикуйте Version», «Разработка забирает конфигурацию через CLI», «Дизайн и код — один источник правды». White Label заменить/дополнить понятным: «одна система — разные бренды».

**Шаг 2. Добавить Developer section.** Getting Started · Supported platforms · Installation · CLI usage · пример компонента · props/states · ссылки на GitHub/пакеты. CLI и перечень свойств — **пометить «contract in progress»**, но показать целевую модель. *(Упирается в решения инженерии.)*

**Шаг 3. Переделать Publish Result.** После публикации — блок «Что произошло»: Version создана · Published Configuration сохранена · участники получили уведомление · в прод ничего не раскатилось · разработчик забирает через CLI · snapshot валидации сохранён в версии. Это снимает главный страх.

**Шаг 4. Улучшить Builder sidebar.** Tooltip на каждую иконку · hover/focus state · подпись активного раздела · внятные иконки. Лучше — expand-on-hover rail.

**Шаг 5. Добавить visual diff.** В Changes и Publish: swatch before/after · значение токена before/after · preview компонента before/after · затронутые компоненты · «к источнику».

**Шаг 6. Добавить Palette view в редактор темы.** Слой: Brand / Neutral / Status палитры + Light/Dark значения, и только затем semantic tokens. Закрывает главный UX-разрыв дизайнера.

**Шаг 7+.** Документация компонентов как Storybook · персонализация Releases · формулировка Rollback · реальные URL/навигация браузера.

## Связь с текущей работой над прототипом

- **Производительность/анимации** (backend отметил задержки) — совпадает с уже идущей оптимизацией hero (dpr-cap 1.5, пауза анимаций вне экрана, 32 карточки, нативный rAF). Замечание валидирует направление; стоит проверить на слабом железе (Intel).
- **Навигация браузера** (frontend: Back/новая вкладка) — упирается в SPA без реальных маршрутов. Рядом с недавним разделением сессии Портал/Builder, но запрос шире: нужны настоящие URL разделов.
- **Стрелка «назад на портал» в Builder** — уже добавлена; косвенно помогает, но корень (подписи/тултипы разделов) ещё открыт (шаг 4).
- **Позиционирование лендинга** — прямой вход в текущую работу над первым экраном (шаг 1).
