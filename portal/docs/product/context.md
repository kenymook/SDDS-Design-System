# Продуктовая рамка DS Builder

> Статус: Draft  
> Основной пользователь MVP: Designer

## Определение

**DS Builder** — рабочий инструмент внутри SDDS Portal, в котором дизайнер настраивает дизайн-систему проекта, редактирует темы, токены и компоненты, проверяет изменения и публикует версионированную конфигурацию для CLI.

Builder не является только редактором цветов: он управляет полным циклом изменений от настройки до публикации результата.

## Основной сценарий MVP

```text
Designer
  → выбирает Project / Design System / Theme
  → редактирует токены и компоненты
  → проверяет preview и validation
  → публикует изменения
  → публикует Version и Configuration для CLI
```

Разработчик — основной потребитель результата публикации, но не основной пользователь Builder. Он выбирает опубликованную Version и применяет её через CLI.

## Главная продуктовая цель

Дать дизайнеру возможность самостоятельно адаптировать SDDS под продукт и передать разработке воспроизводимый результат, не ломая:

- white-label архитектуру;
- семантические токены;
- компонентную модель;
- доступность;
- дизайн-код паритет;
- историю изменений и публикаций.

## Пользователи и роли

| Тип | Название | Задача |
|---|---|---|
| Primary persona | Designer | Редактирует дизайн-систему, проверяет и публикует изменения |
| Consumer | Developer | Получает Version и Configuration через CLI |
| Access role | Owner | Управляет проектом, участниками и доступами |
| Access role | Editor | Редактирует и публикует в Builder |
| Access role | Viewer | Просматривает актуальное состояние без редактирования |

Персона пользователя и роль доступа — разные понятия. Например, Designer может иметь роль `Editor` в одном проекте и `Viewer` в другом.

## Бизнес-проблемы

| Проблема | Последствие | Как помогает DS Builder |
|---|---|---|
| Настройки темы меняются вручную | Ошибки и расхождение дизайна с кодом | Управляемый редактор и единая публикация |
| Нет impact analysis | Непонятно, что затронет изменение | Preview, usage и affected components |
| Нет прозрачной публикации | Разработке непонятно, какой результат использовать | Version, changelog и CLI contract |
| Не определена модель проектов | Смешиваются продукты и дизайн-системы | Project → Design System → Theme |
| Нет контроля качества | Проблемы обнаруживаются поздно | Validation и Health |

## Уже реализованная база

- редактирование цветовых токенов;
- редактирование шрифтов;
- редактирование скруглений;
- частично добавленные компоненты.

## Что DS Builder не должен делать

- Публиковать результат без истории изменений.
- Скрывать влияние изменения от дизайнера.
- Смешивать данные разных проектов.
- Позволять обходить ограничения токенов и компонентов.
- Требовать обязательного review или approve в MVP.

## Maturity model

| Уровень | Название | Результат |
|---|---|---|
| Level 1 | Basic Editing | Редактирование color/typography/rounding |
| Level 2 | Safe Editing | Diff, validation, impact, component editing |
| Level 3 | Direct Publication | Version, changelog и Published Configuration для CLI |
| Level 4 | Governance | Projects, access control, audit и health |
| Level 5 | Enterprise Management | Интеграции, автоматизация и cross-project возможности |
