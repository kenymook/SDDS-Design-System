# Основные пользовательские сценарии

## Знакомство с системой (Portal)

Посетитель открывает главную страницу без авторизации → первый экран отвечает «что это и зачем» (слоган, метрики) → social proof и витрина реальных продуктов → объяснение архитектуры токенов с визуализацией → live demo, где посетитель сам меняет бренд/режим/размер → блоки DS Builder и Developer handoff ведут к инструментам → финальный CTA направляет в onboarding. С мобильного устройства навигация доступна через бургер-меню.

## Изучение документации (Portal)

Пользователь открывает раздел Documentation без авторизации → выбирает страницу в боковой навигации (foundations или Components) → на странице компонента видит живое превью и свойства → при необходимости переходит «Открыть в Builder».

## Прохождение онбординга (Portal)

Новый участник открывает Onboarding → выбирает трек (Designer или Developer) → проходит шаги по порядку, отмечая выполненные; прогресс сохраняется по каждому треку отдельно → шаги ссылаются на документацию, Builder и релизы.

## Просмотр релизов и обращение в поддержку (Portal)

Пользователь открывает Releases и видит опубликованные Versions проектов и релизы SDDS Core. Для запроса доступа, сообщения о баге, запроса компонента или консультации открывает Support и создаёт запрос; из сценариев сбоя (например, неудачная публикация) форма предзаполняется контекстом, и после отправки доступен возврат к исходному сценарию.

## Вход в DS Builder

Пользователь открывает Portal → переходит в отдельное веб-приложение DS Builder → вводит email/логин и пароль. После входа открывается последний доступный Project, а при первом входе — первый доступный. Projects переключаются через иконки в rail; общей страницы Projects нет. Если доступных Projects нет, открывается Create Project. Остальные разделы Portal доступны без авторизации.

## Создание Project

Owner создаёт Project, указывает название и при необходимости сразу приглашает нескольких пользователей как Editor или Viewer. Создатель получает роль Owner, приглашённые memberships относятся только к новому Project, действия записываются в audit log.

## Создание Design System и Theme

Owner открывает конкретный Project и создаёт в нём Design System, указывая название. Платформа не выбирается: каждая Design System поддерживает все платформы. Затем Owner или Editor создаёт Theme, указывает название и стартовую палитру. Режимы Light и Dark входят в Theme автоматически.

## Редактирование tokens

Editor изменяет colors/sizes/fonts, видит old/new, preview, validation и affected entities. Цвет выбирается через color picker (Custom/Library) или вводом hex. Изменение автоматически попадает в Changes; черновики каждого пользователя изолированы.

## Редактирование Component

Editor изменяет разрешённые свойства компонента. Locked properties недоступны, restricted properties сопровождаются объяснением ограничения. Изменение попадает в Changes.

## Просмотр Changes

Editor видит изменения, автора, validation issues и affected entities. Draft-изменение можно revert.

## Прямая публикация

Editor проверяет changes и issues, указывает version и changelog, подтверждает риски и публикует без review/approve.

## Публикация Version

После публикации Builder сохраняет неизменяемую Published Configuration, создаёт Version и уведомляет участников Project (in-app + email). При ошибке сохранения публикация не завершается: показываются error class, причина и request id, доступны Retry и создание Support-запроса с контекстом; draft-изменения не теряются.

## Сравнение и восстановление Version

Editor открывает Version details, видит diff с предыдущей опубликованной Version и при необходимости выполняет «Восстановить как draft»: отличающиеся значения попадают в Changes с подтверждением и публикуются стандартным способом.

## Получение результата разработчиком

Developer выбирает Version и запускает CLI. CLI получает Published Configuration и генерирует target files. Команды, авторизация и способ получения конфигурации пока TBD.

## Управление доступом

Owner открывает Settings конкретного Project и управляет его Members и ролями. Приглашение не даёт доступа к другим Projects Personal Space. Последнего Owner нельзя удалить или понизить. Все изменения доступа логируются.

## Уведомление о приглашении

После приглашения membership действует сразу. Пользователь получает email и in-app уведомление; при следующем входе видит toast, badge на колокольчике и индикатор нового Project в rail. Уведомление открывает Project за один клик. Изменение роли и удаление доступа используют тот же канал уведомлений.
