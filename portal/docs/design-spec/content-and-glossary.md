# Контент и глоссарий

## Язык UI

- **Русский текст + английские названия сущностей.** Интерфейсные фразы — по-русски; сущности и статусы модели — в оригинале: Project, Design System, Theme, Token, Changes, Version, Published Configuration, Draft, Published, Owner/Editor/Viewer.
- Тон — деловой, короткий, без канцелярита. Кнопки — глагол действия («Отправить», «Опубликовать», «Записаться на демо»).
- Честность статуса: то, что в проработке (CLI-контракт, AI/MCP-бета), помечается явно.

## Глоссарий сущностей

| Термин | Значение |
|---|---|
| Project | Изолированное пространство продукта/команды; граница данных, доступа, настроек, audit (роль «тенанта») |
| Project Membership | Связь пользователь↔проект с ролью (owner/editor/viewer) |
| Design System | Набор компонентов и тем внутри проекта |
| Theme | Значения токенов поверх SDDS Base; хранит tokens, modes, subthemes, changes, versions |
| Token | Именованное значение (роль, а не литерал): Color, Typography, Shadow, CornerRadius, Blur |
| SubTheme | Подтема внутри темы |
| Changes | Черновиковые изменения с diff old→new и impact |
| Version | Неизменяемая опубликованная конфигурация с номером и changelog |
| Published Configuration | Снапшот, который CLI отдаёт продукту по номеру Version |
| Draft / Published | Статус темы: черновик / опубликовано |
| Validation issue | Проверка (Warning/Error/Critical), не блокирует публикацию |
| Health | Сводка issues и покрытия |

## Роли (термины UI)

| Термин | В UI |
|---|---|
| Owner / Editor / Viewer | Роли в конкретном проекте (из membership) |
| Designer / Developer | Продуктовые роли только в онбординге; в модель доступа не входят |

## Ключевые тексты-якоря (главная)

- Слоган: «Компоненты готовы. Токены настроены. Займитесь продуктом.»
- Подзаголовок: «Дизайн-система, на которой уже выросло больше десятка продуктов — каждый со своим лицом.»
- Принципы: White-label · Light и Dark из коробки · 4 контекста темы · Единые словари системы.
- CTA: «Записаться на демо» (конверсия) / «Узнать больше» (нарратив).

## Правило

Финальные тексты берём из прототипа как утверждённые формулировки; новые тексты выдерживаем в этом тоне и терминологии.
