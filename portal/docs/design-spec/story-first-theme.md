# Стори нового пользователя: первая тема — от входа до публикации

Сквозной сценарий дизайнера, который впервые попал в DS Builder: создаёт Project и Design System, заводит Theme со стартовой палитрой, настраивает палитру, цветовые токены с контекстами, типографику, скругления и компоненты — и публикует первую Version.

**Зачем документ:** база для задач юзертеста раунда 2 (каждая стори — готовое тестовое задание), чек-лист покрытия для макетирования key screens в S15 и рамка для онбординга «первый сценарий за руку» (уровень 3 лестницы качества).

**Персона:** Аня, продуктовый дизайнер. В команде появился SDDS, у неё есть доступ в портал, в Builder она впервые. Роль после создания проекта — Owner.

Статусы: ✅ есть в прототипе · ⚠️ есть с оговоркой · ⛳ зафиксировано решением, в прототипе пока иначе.

---

## Эпик 1. Вход и пустой старт

### С1. Первый вход
**Как** новый дизайнер, **я хочу** войти в Builder по почте или логину, **чтобы** попасть в своё рабочее пространство.

Критерии приёмки:
- публичные разделы портала доступны без входа; логин требуется только для Builder — и экран входа прямо говорит об этом;
- после входа я попадаю в «мои проекты и доступные мне».

Статус: ✅ (экран «Авторизация» объясняет границу входа; тестовые учётки подписаны).

### С2. Новичок без проектов
**Как** дизайнер без единого проекта, **я хочу** увидеть понятный стартовый экран, **чтобы** выбрать: создать проект или запросить доступ к чужому.

Критерии приёмки:
- заглушка builder-home с первичным действием «Создать проект» и вторичным «Запросить доступ»;
- меня не бросает сразу в форму создания без объяснений.

Статус: ⛳ решение зафиксировано (журнал решений, «Стартовый экран и сессия»), прототип пока редиректит в create-project.

---

## Эпик 2. Создание сущностей

### С3. Создать Project
**Как** дизайнер, **я хочу** создать Project с названием и описанием, **чтобы** получить границу данных и доступа для своей команды.

Критерии приёмки:
- поля: название (обязательно), описание;
- прямо в форме можно пригласить людей и назначить им Editor/Viewer — необязательный шаг;
- кнопка честно говорит, что произойдёт: «Создать и стать Owner».

Статус: ✅ (форма «Создать Project» с блоком приглашений).

### С4. Создать Design System
**Как** Owner, **я хочу** создать Design System внутри проекта, **чтобы** в ней жили темы продукта.

Критерии приёмки:
- платформу выбирать не нужно — все DS поддерживают все платформы (решение);
- после создания я вижу карточку DS в списке проекта.

Статус: ✅.

### С5. Создать Theme со стартовой палитрой
**Как** дизайнер, **я хочу** при создании темы выбрать стартовую палитру, **чтобы** не начинать с пустых значений.

Критерии приёмки:
- варианты: SDDS Base (по умолчанию), Contrast, Custom;
- Custom раскрывает редактор: пресеты Neutral/Brand/Warm и ручные HEX для Primary / On primary / Background / Text с валидацией формата;
- форма объясняет: палитра задаёт начальные Colors, дальше всё редактируется внутри темы;
- палитра выбирается у темы, а не у Design System (решение).

Статус: ✅ (форма «Создать Theme», блок «Стартовая палитра»).

---

## Эпик 3. Первое знакомство с редактором

### С6. Ориентация в рабочем пространстве
**Как** новичок, **я хочу** с первого экрана редактора понимать, где я и что могу, **чтобы** не заблудиться.

Критерии приёмки:
- рейл слева: Health · Palette · Colors · Typography · Corner radius · Components · Changes · Versions · Documentation — с тултипами;
- топбар одинаков на всех вкладках: крошки, по центру версия · статус · роль, справа Reset и «Publish · 0»;
- пока изменений нет, Publish задизейблен и счётчик показывает 0.

Статус: ✅ (единый топбар, фирменные иконки рейла).

---

## Эпик 4. Настройка палитры и цвета

### С7. Палитра бренда каскадом
**Как** дизайнер, **я хочу** поменять базовый цвет бренда в одном месте, **чтобы** семантические токены пересчитались сами.

Критерии приёмки:
- вкладка Palette: рампы Brand (ступени 50–900), Neutral, Status;
- правка базовой ступени 500 пересчитывает остальные ступени и Dark-режим;
- блок «Куда дотянется каскад» показывает цепочку brand-500 → color.primary → компоненты с числом вхождений;
- Neutral и Status фиксированы системой — и это объяснено на экране.

Статус: ✅ (Palette view, запрос из юзертестов раунда 1).

### С8. Цветовой токен: значения режимов
**Как** дизайнер, **я хочу** видеть и править Light/Dark значения токена рядом, **чтобы** тёмная тема не была отдельным проектом.

Критерии приёмки:
- дерево токенов по группам (Text & Icons, Surfaces, Outlines, BG, Data…) с чипами Light/Dark-цвета у каждой строки;
- борд свотчей всей палитры темы на канвасе;
- инспектор: описание токена, строки Light и Dark со свотчем, именем значения и прозрачностью;
- цвет можно выбрать пипеткой или ввести значение.

Статус: ✅.

### С9. Context: как токен ведёт себя на разных поверхностях
**Как** дизайнер, **я хочу** чтобы контексты (OnDark, OnLight, Inverse) наследовались из моих режимов автоматически, **чтобы** не заполнять их руками.

Критерии приёмки:
- OnDark берёт значения Dark-режима, OnLight — Light, Inverse меняет их местами;
- правка Default сразу видна в контекстах;
- ручная правка контекста рвёт связь (иконка меняется), клик по иконке возвращает наследование и сбрасывает ручные значения;
- у блока есть подсказка, объясняющая механику.

Статус: ✅.

### С10. Живой пример вместо догадок
**Как** дизайнер, **я хочу** переключить канвас на живой компонент, **чтобы** увидеть свои draft-значения на настоящем Button или поле ввода.

Критерии приёмки:
- селект вида канваса: Palette (борд) / Button / Text Field / Typography; More честно помечен «в проработке»;
- пример собран из текущих draft-значений и меняется при правках.

Статус: ✅.

## Эпик 5. Типографика, скругления, компоненты

### С11. Типографика
**Как** дизайнер, **я хочу** задать шрифт и базовый кегль, **чтобы** увидеть всю шкалу целиком.

Критерии приёмки:
- вкладка Typography: борд-шкала Display/Heading/Title/Body/Caption, вычисленная от body;
- инспектор в том же строчном стиле, что у цветов, с превью строки;
- правка кегля пересчитывает шкалу.

Статус: ✅.

### С12. Скругления в рамках ограничений
**Как** дизайнер, **я хочу** менять скругление в разрешённых пределах, **чтобы** не сломать систему.

Критерии приёмки:
- значение и допустимый диапазон (0–24) показаны рядом; выход за пределы — явная ошибка;
- борд показывает скругление на поверхностях разного размера (Control, Card, Modal).

Статус: ✅ («редактируем в рамках ограничений» — решение).

### С13. Свойства компонентов по политике
**Как** дизайнер, **я хочу** видеть, какие props компонента можно менять, **чтобы** не упираться в запреты без объяснений.

Критерии приёмки:
- у свойства видна политика: editable / restricted / locked;
- restricted и locked объяснены, а не просто задизейблены.

Статус: ⚠️ политика есть; полный перечень изменяемых props — открытое решение уровня 2 (держит инженерия).

---

## Эпик 6. Качество и публикация

### С14. Проверки не блокируют, а помогают
**Как** дизайнер, **я хочу** видеть срез качества темы, **чтобы** починить проблемы до публикации — но не быть заблокированным.

Критерии приёмки:
- Health: общий score, Critical/Errors/Warnings, покрытие;
- вкладка Accessibility в редакторе показывает контрастные пары; у замечания есть исправление в один клик;
- замечания не блокируют Publish (решение) — но видны в чек-листе публикации.

Статус: ✅.

### С15. Changes: что именно улетит
**Как** дизайнер, **я хочу** просмотреть список своих правок перед публикацией, **чтобы** понимать, что войдёт в версию.

Критерии приёмки:
- список изменений с diff «было → станет» и severity;
- undo/redo и сброс (точечный, раздела, всех) с подтверждением;
- черновик — мой личный: другие видят опубликованное, а не мои правки (решение);
- счётчик изменений виден постоянно на кнопке Publish.

Статус: ✅.

### С16. Публикация
**Как** дизайнер, **я хочу** опубликовать версию сам, без согласований, **чтобы** передать изменения разработке.

Критерии приёмки:
- экран Publish: чек-лист готовности, следующая semver-версия, поле changelog;
- прямая публикация: review и approve не требуются (решение);
- если черновик отстал от новой опубликованной версии — сначала предлагается rebase, а не молчаливая перезапись;
- при сбое — понятная ошибка с кодом и обращением в поддержку с заполненным контекстом (решение).

Статус: ✅.

### С17. «Что произошло»
**Как** дизайнер, впервые нажавший Publish, **я хочу** явное объяснение результата, **чтобы** не гадать, куда улетели изменения.

Критерии приёмки — экран результата говорит:
- Version создана и неизменяема;
- Published Configuration сохранена;
- уведомления участникам ушли;
- в прод ничего не раскатилось само — разработчик забирает версию через CLI;
- видно, чем версия отличается от предыдущей.

Статус: ✅ (добавлено по итогам интервью раунда 1 — respondents не понимали, «куда улетают изменения»).

### С18. Жизнь после публикации
**Как** дизайнер, **я хочу** видеть историю версий и уметь откатываться, **чтобы** ошибка не была фатальной.

Критерии приёмки:
- Versions: список с датами, статусами, отличиями от предыдущей;
- откат — «Восстановить как черновик» с подтверждением; история не переписывается;
- restore спрятан, когда отличий нет (решение по итогам раунда 1).

Статус: ✅.

## Прогон по прототипу — 22.07.2026

Автоматизированный сквозной прогон всех стори с чистого состояния: вход → Project «Retail Bank» → DS «Retail DS» → Theme «Retail Light» (Custom-палитра, пресет Brand) → правка primary → Typography 18px → radius с проверкой диапазона → Publish → «Что произошло» → Versions.

**Итог: 17 из 17 проверяемых стори проходят** (С2 — новичок без проектов — не проверяется на демо-учётках, статус ⛳ без изменений).

Находки прогона:
1. **Semver новой темы считается от чужих версий.** Тема создана как v0.1.0 Draft, но Publish предложил 1.1.0 — `nextVersion()` брал максимум по всем Published-версиям вообще, а не по версиям этой темы. **Исправлено 22.07.2026:** версия у каждой темы своя — следующая считается от текущей версии темы (той же, что в топбаре): v1.4.0 → 1.5.0, новая v0.1.0 → 0.2.0.
2. Кастомные радио палитры (стартовая палитра, пресеты Custom) не кликались без реального курсора — стилизованный input был схлопнут в точку, фокус с клавиатуры невидим. **Исправлено 22.07.2026:** input накрывает карточку целиком (кликабелен, фокусируем, виден автоматике), стрелки переключают выбор, у сфокусированной карточки видимое кольцо.

## Как использовать в раунде 2

Минимальный прогон теста — четыре задания без подсказок модератора:
1. «Создайте проект и тему с палитрой вашего бренда» (С2–С5);
2. «Сделайте primary-цвет фирменным и проверьте, как он выглядит на кнопке» (С7–С10);
3. «Поменяйте шрифт и скругление» (С11–С12);
4. «Опубликуйте изменения и объясните своими словами, что произошло» (С15–С17).

Критерий успеха раунда — из лестницы качества: новый пользователь проходит сценарии без единого вопроса; проверка С17 — респондент сам произносит «разработчик заберёт через CLI, в прод ничего не уехало».
