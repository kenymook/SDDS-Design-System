# S14 — план и статус дизайн-задач: Builder и Portal

**Спринт:** 9 – 22 июля 2026 · продолжение S13.
Здесь — задачи S14 по **DS Builder** и **SDDS Portal** с привязкой к уже готовым материалам этого пространства. Трек **Design 2.0 (пересборка 6 компонентов: Toast, Notification, TextField, Select, Button, Tabs + шаблон документации)** ведётся отдельно и в это пространство не входит — по договорённости.

По итогам актуализации S14 закрывает не только discovery, но и **product foundation для DS Builder 1.0**: user flow, авторизация, RBAC UI, модель Project / Design System / Theme и поддержка градиентов зафиксированы как база для UI-отрисовки S15.

## Цель S14

Перейти от scope и первичного аудита S13 к структурным решениям по Builder и Portal. S14 — **не спринт финальной отрисовки**: финальный UI начинается с S15. Задача — закрыть продуктовую и UX-основу, чтобы с S15 ускориться к key screens, UI direction и handoff.

## Приоритеты (Builder/Portal-часть)

- **P0:** DS Builder MVP User Flow · DS Builder Authentication Flow · RBAC UI Design · Project & Design System Management UX · Gradient Support in Builder · Support/Duty
- **P1:** SDDS Portal MVP IA · список вопросов для PM/dev/backend · подготовка решений для S15
- **P2 (не в S14):** детальный UI портала · полный onboarding · расширенные состояния Builder · продвинутые RBAC-сценарии

## Задачи и что уже готово

Ключевое: задачи опираются на материалы, собранные в этом пространстве (доки + работающий прототип + доска экранов в Figma). Ниже — маппинг: что уже закрыто, что становится входом для S15.

### [S14] DS Builder Authentication Flow — Руслан · P0

Сценарии входа в Builder, проверки доступа, выхода, ошибок и перехода из Portal в рабочее пространство.

| Критерий | Что уже есть | Что осталось |
|---|---|---|
| Путь Portal → Auth → Builder | [Спецификация авторизации](../deliverables/01-builder-auth-spec.md) + прототип: CTA «Открыть DS Builder» ведёт на вход и далее в рабочее пространство | Перенести в финальный UI в S15 |
| Аккаунт и роль пользователя | Роль не выбирается при входе; подтягивается из Project Membership | Уточнить технический провайдер входа с безопасностью |
| Нет аккаунта / нет доступа | Вход ведёт к «Запросить доступ» через поддержку | Отрисовать финальное состояние |
| Ошибка входа / выход / сессия | Основные правила описаны в auth-spec и решениях | Детализировать визуальные состояния в макетах |

**Статус:** ✅ Done на уровне UX-спецификации. В S15 остаётся визуальная отрисовка экранов входа и access states.

### [S14] RBAC UI Design — Руслан · P0

UI-паттерны ограничений доступа: full access, read-only, no access/403, disabled, hidden.

| Критерий | Что уже есть | Что осталось |
|---|---|---|
| Full access / read-only / no access | [Правила RBAC](../deliverables/03-rbac-ui-rules.md) + **реализовано в прототипе** и снято на доску: R1 «Нет доступа», R2 «Project глазами Viewer», редактор/Publish в режиме Viewer | Свести в таблицу паттернов «роль × действие» |
| Disabled / hidden actions | В прототипе есть оба паттерна (disabled Publish для Viewer c объяснением; скрытые пункты настроек) + принцип из решений: «Viewer видит, почему недоступно, и может запросить доступ» | ✅ **Решено:** действия (публикация/создание/откат) — **hidden**; объяснение баннером на экране; поля ввода — режим «только просмотр» (см. [журнал решений](decisions-log.md#редактирование-и-качество)) |
| Вопросы к backend | В журнале решений: модель permissions, вход (Sber ID / корпоративный) | Оформить списком к Builder Sync |

**Статус:** ✅ Done на уровне UI-правил. Роли Owner / Editor / Viewer применяются на уровне Project; Design System и Theme наследуют доступы проекта.

### [S14] Gradient Support in Builder — Руслан · P0

Новый функционал редактора: поддержка градиентов в Builder как расширение работы с визуальными параметрами темы.

| Критерий | Что уже есть | Что осталось |
|---|---|---|
| Место в Builder | Градиенты работают внутри редактора темы и color picker / Palette flow | Зафиксировать в финальных UI-макетах |
| Модель значения | Тип градиента, стопы, позиция, цвет и прозрачность | Уточнить формат хранения с разработкой при handoff |
| Поведение | Добавление/выбор стопов, редактирование значения, preview | Описать edge cases в спецификации редактора |
| Связь с темой | Значение сохраняется как часть Theme и попадает в изменения | Проверить отображение diff в Changes |

**Статус:** ✅ Done как реализованный функциональный блок прототипа/Builder; в S15 важно не потерять его в финальной отрисовке.

## Синхронизации

1. **Kick-off** (вся команда) — S14 не ускоряем, ускорение с S15; Builder и 6 компонентов — приоритеты; Portal — только IA и MVP.
2. **Builder Sync** (Руслан, Лиза, PM/dev) — сущности, экраны для S15, роли/permissions, авторизация, градиенты, блокеры UI. *Готовая повестка: открытые вопросы из журнала решений (CLI, props, вход).*
3. **Design 2.0 Sync** (Руслан, Ира, Лиза, Ася) — отдельный трек.
4. **Support Check** (Руслан, Ася).
5. **S14 Review** (вся команда) — flow, auth, RBAC, Project/DS UX, gradients, Portal IA, шаблон, 6 компонентов, вопросы и риски на S15.

## Definition of Done (Builder/Portal-часть)

- по Builder есть понятный user flow ✅ *(база готова, оформить схему)*;
- по авторизации Builder есть UX-спецификация ✅ *(в S15 — финальный UI)*;
- по RBAC есть базовые UI-сценарии ✅ *(база готова, свести таблицу)*;
- по Project & Design System Management есть UX draft ✅ *(база готова, схема связей)*;
- поддержка градиентов в Builder зафиксирована ✅ *(в S15 — перенести в UI-макеты и handoff)*;
- по Portal есть MVP IA ✅ *(база готова, оформить схемой)*;
- по каждому направлению есть список вопросов на S15 — собрать из журнала решений и отчёта по интервью;
- команда понимает, какие экраны переходят в ускорение с S15 — это напрямую покрывает **нарезка экранов для макетирования**: [DS BUILDER | PLAYGROUND, страница «СС»](https://www.figma.com/design/3qtcAteqALdFuRL6EvR7qr/DS-BUILDER-%7C-PLAYGROUND?node-id=591-4822) — 63 кадра (25 экранов + состояния, попапы, ошибки, роли, мобильные) с аннотациями «гипотеза сработала/провалилась/исправлено» по итогам юзертестов.

## Риски S14 и как наши материалы их снижают

| Риск | Митигация из плана | Чем помогает готовое |
|---|---|---|
| Слишком ранний уход в UI | В S14 только схемы/драфты | Прототип и доска уже дают «больше чем схемы» без затрат на отрисовку — материал для решений, не финальный UI |
| Portal конкурирует с Builder | Портал ограничен IA и scope | IA фактически готова — портал почти не потребует capacity |
| RBAC заблокирует Builder | Фиксируем UI-паттерны, permissions — в вопросы | Паттерны уже реализованы и сняты; ждать backend не нужно |
| Новый функционал градиентов потеряется при перерисовке | Добавляем Gradient Support в S14 Done и S15 handoff | Функционал уже реализован в прототипе/Builder, остаётся перенести в финальные макеты |
| Support съест capacity | Отдельная задача, оценка объёма | Метрика «стоимость поддержки» заведена в экономике |
