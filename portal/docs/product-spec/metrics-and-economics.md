# Метрики, discovery и экономика

> Черновик для согласования с PM. Delivery-документы отвечают «что строим и когда»; этот — «как поймём, что работает, для кого и что это стоит». Детализирует раздел «Как поймём, что получилось» в [бизнес-требованиях](business-requirements.md#6-как-поймём-что-получилось).

## 1. Поведенческие метрики

**North Star: команды, регулярно публикующие версии тем через Builder** — ≥1 публикация в месяц силами самой команды, без участия команды SDDS.

### Воронка активации команды

| Шаг | Событие в жизни команды |
|---|---|
| A0 | Заявка отправлена / доступ выдан |
| A1 | Первый вход в Builder |
| A2 | Онбординг завершён (все шаги трека) |
| A3 | Создан первый Project |
| A4 | Настроена тема — первое изменение токена в draft |
| A5 | Опубликована первая Version |
| A6 | Version получена разработчиком через CLI — handoff состоялся |
| R1 | Команда вернулась в течение 14 дней |
| R2 | **Вторая публикация** — критерий «10/10» из [лестницы качества](quality-ladder.md) |

### Метрики по воронке

| Метрика | Определение | Ориентир пилота |
|---|---|---|
| Команды, начавшие пользоваться | уникальные команды с A1 / выданных доступов | 100% за 2 недели |
| Завершили онбординг | A2 / A1 | ≥ 60% |
| Создали проект | A3 / A1 | ≥ 80% |
| Настроили тему | A4 / A3 | ≥ 80% |
| Опубликовали | A5 / A4 | ≥ 70% |
| Handoff состоялся | A6 / A5 | ≥ 90% — иначе Builder «песочница», а не источник правды |
| Вернулись повторно | R1; R2 | R1 ≥ 50%; R2 ≥ 40% |

**Drop-off:** смотрим наибольшее падение между соседними шагами. Кандидаты по данным юзертестов: **A4** (ментальная модель редактора — частично закрыто Palette view, проверить в раунде 2) и **A5→A6** (CLI не зафиксирован — держит инженерия).

### Здоровье (анти-метрики)

- Доля сбоев публикации; доля сбоев с успешным повтором (черновик не потерян — обещание версии).
- Публикации с непогашенными validation issues.
- Обращений в поддержку на команду в месяц — должно падать (связка с экономикой, §3).
- Частота отката версий (всплеск = проблема качества правок или UX).

### События

**Уже в прототипе** (Vercel Analytics, `track()`): `view:<раздел>`, `cta:записаться-на-демо`, `cta:открыть-ds-builder`, `cta:узнать-больше`, `заявка-отправлена` (+тип).

**Добавить для воронки:** `builder_login`, `onboarding_step` / `onboarding_completed`, `project_created`, `ds_created`, `theme_created`, `token_changed` (первое = A4), `palette_cascade_used`, `publish_started`, `publish_success` / `publish_failed` / `publish_retry`, `cli_copied`, `version_restored`.

**Важно для прода:** события должны нести идентификатор команды/проекта — иначе «сколько команд» не посчитать. В прототипе достаточно веб-аналитики; в проде — продуктовая аналитика с воронками.

## 2. Discovery и рынок

### Первый сегмент и как он выбран

**Продуктовые команды экосистемы Сбера**, которые: (а) уже строят на SDDS или мигрируют, (б) имеют собственный бренд/тему, (в) имеют в составе и дизайнера, и разработчиков. Выбор не «по размеру рынка», а по **наименьшему расстоянию до ценности**: библиотекой уже пользуются 16+ команд, боль темизации у них уже есть — продавать нужно не дизайн-систему, а только инструмент поверх неё.

### Проблемы, подтверждённые интервью (раунд 1, 07.2026 — 3 респондента)

| Проблема | Доказательство из интервью |
|---|---|
| Ручной перенос токенов дизайн → код; источник правды в Figma | Дизайн-лид: токены в Figma variables + самописный скрипт выгрузки в CSS; «если я что-то поменяю и не скажу разрабам — провисит очень долго» |
| Непрозрачный handoff | «Дизайнер скинул макет» как текущая норма; на тесте никто не понял, «куда улетает» публикация |
| Нет версионирования темы | Git-модель Version/CLI вызвала доверие у обоих разработчиков — «как обновление проекта через Git» |
| Порог входа для разработчика | Все трое искали installation, props, примеры кода |

Полные данные — в [отчёте по интервью](../research/2026-07-07-interviews-round1.md).

### Альтернативы, которыми пользуются сейчас

- **Figma variables + самописные скрипты выгрузки** (подтверждено интервью — так живёт команда дизайн-лида).
- **Хардкод значений в коде** + тикеты на каждое изменение.
- **Запросы в команду дизайн-системы** — ручная поддержка, которая не масштабируется (см. §3).
- **Внешние библиотеки** (респондент сам сравнил с shadcn/ui) — дают компоненты, но не решают мультибренд-темизацию и handoff.

### Почему Builder и почему именно этот MVP

Ценность сформулировал сам респондент: *«токены в одном месте — источник правды и для разработки, и для дизайна… эту штуку я бы продал с удовольствием»*. Выбранный MVP бьёт в подтверждённые проблемы одной цепочкой **Theme → Changes → Version → CLI**: убирает ручной перенос, делает handoff видимым и версионированным. Публикация без согласований — скорость (решение зафиксировано в [журнале](decisions-log.md)).

### Что ещё не доказано — честно

- **Handoff/CLI** протестирован бегло у всех троих — фокус раунда 2.
- Выборка — 3 человека: крупное поймано, тонкое — нет; «палитра» — голос одного дизайнера.
- Экономика ниже — **модель с гипотезами**, не факты.

## 3. Экономика

Прямо: реальных цифр пока нет. Ниже — модель с формулами и план замера «до/после» на пилоте. Данные заполняются с PM и командой поддержки SDDS.

| Величина | Формула | Данные | Как замерим на пилоте |
|---|---|---|---|
| Стоимость текущей поддержки темизации | N запросов/мес × T часов × ставка | *заполнить: статистика тикетов команды SDDS* | Тикеты «до» за 2–3 месяца |
| Экономия self-service | (T вручную − T в Builder) × число правок тем | T в Builder — **минуты**: на юзертестах путь «правка → публикация» все трое прошли за ≤15 мин; T вручную — со слов дизайн-лида цепочка «Figma → сказать разрабам → скрипт → проверка» занимает **дни** | Хронометраж 3–5 реальных правок |
| Стоимость подключения команды | T онбординга × ставка + сопровождение | *заполнить* | Часы команды SDDS на 1-ю vs 3-ю команду пилота |
| Влияние на запуск продукта | время «бренд-тема готова к разработке» | гипотеза: недели → дни | Дата старта пилота → дата первой Version |
| Снижение ручной работы | число ручных выгрузок токенов × время | *заполнить: опрос команд «до»* | События `cli_copied`/`publish_success` «после» |

**Главный экономический аргумент (гипотеза к проверке):** сейчас темизация масштабируется людьми команды SDDS — каждая новая команда добавляет ручную работу. Builder переводит её в self-service: стоимость подключения N-й команды перестаёт расти. Проверка — сравнение часов сопровождения первой и третьей команды пилота.
