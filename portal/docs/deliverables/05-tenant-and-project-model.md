# Тенантная модель и путь от входа до работы

Спринт-задача (дизайнер команды) · [Прототип](https://portal-flow-prototype.vercel.app) → «Открыть DS Builder» (учётки `anna`, `ivan`, `maria`, пароль `demo`).

## 1. О чём задача и где границы

Нужно спроектировать всё, что человек видит **между входом и редактором темы**: своё пространство, выбор проекта, Design System и темы, переключение между ними, изоляцию данных. Проще говоря — как человек добирается до рабочего места и не путается по дороге.

Где задача заканчивается: на входе в саму тему. Что происходит внутри редактора — это уже [границы MVP Builder](02-ds-builder-mvp-scope.md). Права ролей на экранах — [правила RBAC](03-rbac-ui-rules.md). Вход — [спецификация авторизации](01-builder-auth-spec.md).

## 2. Что такое «тенант» у нас

Отдельной сущности «тенант» нет. Роль границы играет **проект (Project)** — это и есть наш тенант: обособленное пространство продукта или команды со своими участниками, ролями, настройками и журналом действий. Формулировка закреплена в архитектуре: проект — граница данных и доступа, и слово «тенант» в интерфейсе мы не используем.

## 3. Как всё устроено по уровням

```text
Пользователь
  └── Участие в проекте (роль в конкретном проекте)
        └── Project                 ← граница тенанта: данные, доступ, настройки, журнал
              └── Design System
                    ├── Компоненты
                    └── Theme         ← вход в рабочее пространство (граница задачи)
                          ├── Токены / режимы / подтемы
                          ├── Изменения
                          └── Версии
```

Один человек может состоять в нескольких проектах с **разными ролями** — где-то Owner, где-то Editor, где-то Viewer.

## 4. Своё пространство (сразу после входа)

Это не отдельная сущность, а просто удобный вид на все проекты человека:

- **Мои проекты** — те, где он Owner;
- **Доступные мне** — куда его пригласили как Editor или Viewer.

Что происходит при входе:
- открывается последний доступный проект; при самом первом входе — первый доступный;
- если проектов нет — показываем **экран-заглушку** (не редирект сразу в create-project): «Создать проект» первичным действием, «Запросить доступ» — вторичным. Так новичок-создатель и новичок-приглашённый видят каждый свой путь.

## 5. Экраны на пути к работе

| Экран | Что на нём | 
|---|---|
| Стартовый экран Builder | Панель проектов (иконки доступных), вход в последний, кнопка «создать» |
| Обзор проекта | Его Design System, участники, сводка по качеству, последние публикации; у Owner — настройки |
| Обзор Design System | Список тем со статусом и версией; создание темы (Owner или Editor) |
| → Тема | Переход в рабочее пространство (редактор) — здесь задача заканчивается |

Отдельной страницы «все проекты» нет — переключаются через панель проектов.

## 6. Контекст, который всегда виден

Пока человек работает, он не должен гадать, где находится. Поэтому наверху всегда строка:

```text
Project / Design System / Theme / версия / статус
пример: SberBusiness / CRM DS / Light Theme / v1.4.0 / черновик
```

## 7. Правила переключения и изоляции (в этом и есть суть тенанта)

- **Контекст всегда виден** — текущие проект, система и тема.
- **Права пересчитываются при смене проекта** — роль берётся из участия в новом проекте.
- **Предупреждаем о несохранённом** — если при переходе есть незакоммиченные правки, спрашиваем подтверждение.
- **Данные проектов не смешиваются** — поиск, история и публикации одного проекта не пересекаются с другими.
- **Черновики у каждого свои** — пока не опубликовал, правки видит только их автор; команде достаётся лишь опубликованное.

## 8. Сущности и их поля (для передачи разработке)

- **Project:** идентификатор, название, описание, статус (активен/архивирован), владелец, список Design System, участники.
- **Участие в проекте:** пользователь, проект, роль (owner/editor/viewer), статус (активно/приглашён/отключён).
- **Design System:** идентификатор, проект, название, компоненты, темы, текущая версия, ссылка на репозиторий и на библиотеку в Figma/Pixso (если есть), статус.
- **Theme:** принадлежит одной Design System; хранит токены, режимы, подтемы, изменения, версии. Значение по умолчанию унаследовано; если его переопределили — это видно и попадает в изменения.

## 9. Пустые и пограничные состояния (обязательно отрисовать)

| Ситуация | Что показываем |
|---|---|
| Проектов нет | Пустой экран + создать проект / запросить доступ |
| В проекте нет Design System | Пустой экран + создать Design System (только Owner) |
| В Design System нет тем | Пустой экран + создать тему (Owner или Editor) |
| Человек — Viewer в проекте | Просмотр + баннер «только для просмотра» + запрос прав |
| Уходит с несохранёнными правками | Диалог с подтверждением |

## 10. Что ещё не решено

1. Нужен ли обучающий проект-песочница для новичков (уже стоит как открытый вопрос в архитектуре).
2. Как показать, что у человека разные роли в разных проектах (индикатор роли прямо в панели проектов?).
3. С какого числа проектов панель из иконок пора заменить полноценным списком с поиском.
4. Нужен ли уровень «организация → проект» над проектами, если границ тенанта станет две. Сейчас модель одноуровневая: проект и есть тенант.
