# Что входит в MVP DS Builder

Спринт-задача 2 · [Прототип](https://portal-flow-prototype.vercel.app) → «Открыть DS Builder».

## 1. Что такое DS Builder

Это рабочий инструмент дизайнера внутри портала. В нём настраивают тему под бренд прямо на токенах, проверяют изменения и публикуют версию, которую разработчик забирает через CLI. Без кода и без долгих согласований — в этом вся идея.

## 2. Принципы, которые задают рамку

- **Сначала дизайнер** — весь основной сценарий вылизан под него.
- **Всегда в контексте** — любое действие происходит внутри связки Project → Design System → Theme.
- **Безопасно, но не мешая** — изменения видны, обратимы и проверяемы; проверки предупреждают, но не блокируют публикацию.
- **Через токены** — настройка идёт только управляемыми токенами и разрешёнными свойствами.
- **Дизайн и код — одно** — у обоих один результат и одна версия.

## 3. Основной сценарий целиком

```text
Открыть Project / Design System / Theme
  → поправить токены и компоненты
  → посмотреть предпросмотр
  → пройти проверки (они не блокируют)
  → опубликовать (без согласований)
  → получить версию и конфигурацию
  → разработчик забирает версию через CLI
  → следить за качеством, историей, версиями
```

## 4. Что берём в MVP (обязательное)

Каждый пункт уже показан в прототипе — рядом указан экран.

| Область | Что должно быть | Экран в прототипе |
|---|---|---|
| Своё пространство | Вход в доступные проекты и последние Design System | стартовый экран Builder |
| Модель проектов | Project → Design System → Theme | обзоры проекта и системы |
| Доступ | Роли Owner / Editor / Viewer из участия | см. [правила RBAC](03-rbac-ui-rules.md) |
| Обзор темы | Статус, версия, изменения, качество | строка контекста + обзор темы |
| Правка токенов | Цвет, градиенты, шрифты, скругления + picker | редактор |
| Правка компонентов | Разрешённые свойства с ограничениями | редактор компонентов |
| Предпросмотр | Живая проверка темы и компонентов | панель предпросмотра |
| Изменения | Старое → новое, что затронуто, откат, отмена/повтор | экран изменений |
| Проверки | Предупреждения, контраст по WCAG, готовые исправления | инспектор + экран качества |
| Публикация | Дизайнер публикует сам, без согласований | экран публикации |
| Передача в разработку | Конфигурация по номеру версии | итог публикации + CLI |
| История | Версии, описания, сравнение, откат через «Восстановить как черновик» | версии |
| Сбой публикации | Ошибка с кодом и повтором, черновик не теряется | итог публикации (ошибка) |
| Уведомления | Участникам — о публикации и доступе | всплывающие уведомления |

> Экраны «от входа до рабочего пространства» (стартовый экран Builder, проект, Design System) подробно спроектированы в [модели проектов и дизайн-систем](05-project-and-design-system-model.md). Здесь они зафиксированы как входящие в MVP; их детальный дизайн — там.

## 5. Приоритеты (P0/P1/P2) и рамка Q3

> Черновик для согласования с PM и разработкой (это прямо в критериях готовности задачи). Приоритеты опираются на основной сценарий и зависимости; кварталы — ориентир, а не жёсткое обязательство.

**Что значат уровни:**
- **P0 — критический путь.** Без этого Builder не работает как продукт: весь путь дизайнера от выбора темы до публикации версии.
- **P1 — важное для ценности и безопасности.** Сильно повышает пользу; пилот можно ненадолго запустить без части P1.
- **P2 — желательное.** Спокойно переносим в Q4, если не влезает.

| Область (из обязательного) | Приоритет | Квартал | Зависит от / риск |
|---|---|---|---|
| Своё пространство, панель проектов | P0 | Q3 | — |
| Модель Project → Design System → Theme | P0 | Q3 | — |
| Роли Owner / Editor / Viewer | P0 | Q3 | — |
| Правка токенов (цвет/градиенты/шрифты/скругления) + предпросмотр | P0 | Q3 | — |
| Изменения: различия, откат, отмена/повтор | P0 | Q3 | — |
| Прямая публикация + версия | P0 | Q3 | — |
| Передача через CLI | P0 | Q3 (риск сдвига) | договорённость по CLI — открыта |
| Проверки: замечания, контраст, готовые исправления | P1 | Q3 | — |
| Сбой публикации + повтор | P1 | Q3 | — |
| Правка компонентов | P1 | Q3 (риск сдвига) | перечень свойств — открыт |
| Качество (Health) | P1 | Q3 → Q4 | после базовых проверок |
| Версии: сравнение и откат как черновик | P1 | Q3 → Q4 | после базовой публикации |
| Уведомления о публикации и доступе | P2 | Q4 | — |
| Настройки проекта/системы, журнал действий | P2 | Q4 | — |

**Не входит в Q3 (остаётся в MVP, но переносим в Q4):** качество и версии-с-откатом, если не успеваем после публикации; уведомления; администрирование, настройки и журнал. Полностью вне MVP — раздел 6.

**Что может сдвинуть сроки:** два открытых решения держат часть P0/P1 — договорённость по CLI (передача результата) и перечень изменяемых свойств компонентов. Их важно закрыть в начале Q3.

## 6. Что осознанно не берём

Не потому что «плохо», а потому что сначала должна заработать основа.

| Что | Почему пока нет |
|---|---|
| Согласование и ревью публикаций | На этом этапе не нужно — дизайнер публикует сам |
| Наследование настроек между проектами | Слишком сложно для первого шага |
| Продвинутая аналитика | После базового качества |
| Автодоставка собранных файлов | Это забота CLI, а не Builder |
| Генерация тем по описанию (ИИ) | Заявлено как направление, но не обещание MVP |
| Полная договорённость по CLI | В проработке (в интерфейсе так и помечено), фиксируется инженерией |

## 7. Как поймём, что MVP получился

Дизайнер может:
1. выбрать проект, Design System и тему;
2. поправить токены, градиенты и разрешённые свойства компонентов;
3. увидеть различия, предпросмотр и влияние правок;
4. увидеть замечания, но не застрять из-за них;
5. опубликовать без согласований;
6. получить версию и конфигурацию для разработчика;
7. понять, кто и когда опубликовал результат.

## 8. От чего зависим и что не решено

- **Вход** — см. [спецификацию авторизации](01-builder-auth-spec.md); через что именно логинятся, пока не выбрано.
- **CLI** — команды и формат конфигурации в проработке (и в прототипе, и на лендинге это честно помечено).
- **Свойства компонентов** — какие можно менять, какие ограничить, какие закрыть — открытое решение команды системы.
- **Согласование боевых тем** — вопрос всплыл после прогонов с пользователями; в текущий MVP не входит.
