# Роли и права — как это выглядит в интерфейсе

Спринт-задача 3 · [Прототип](https://portal-flow-prototype.vercel.app): войдите как `maria` (Viewer), `ivan` (Editor) или `anna` (Owner), пароль `demo`.

## 1. Как устроены роли

Права даются человеку не «вообще», а внутри конкретного проекта. Поэтому один и тот же человек может быть Owner в своём проекте и Viewer в чужом. Роль при входе не выбирают — она подтягивается из участия в проекте.

Ролей три:

| Роль | Что делает |
|---|---|
| Owner | Управляет проектом: участники, Design System, настройки, публикация |
| Editor | Редактирует темы, проверяет и публикует |
| Viewer | Смотрит состояние, версии и результаты, но не редактирует |

Разработчик — это потребитель результата через CLI, а не отдельная роль. Если ему нужен доступ к проекту, хватает Viewer. И ещё раз: профессия (дизайнер, разработчик) и роль доступа (Owner, Editor, Viewer) — разные оси.

## 2. Кто что может (главная таблица)

| Действие | Viewer | Editor | Owner |
|---|:--:|:--:|:--:|
| Смотреть проект и Design System | ✓ | ✓ | ✓ |
| Смотреть темы и компоненты | ✓ | ✓ | ✓ |
| Смотреть изменения, качество, версии | ✓ | ✓ | ✓ |
| Создать Design System | ✗ | ✗ | ✓ |
| Создать тему | ✗ | ✓ | ✓ |
| Редактировать токены | ✗ | ✓ | ✓ |
| Редактировать свойства компонентов | ✗ | ✓ | ✓ |
| Запускать проверки | ✗ | ✓ | ✓ |
| Откатывать свои правки | ✗ | ✓ | ✓ |
| Публиковать версию | ✗ | ✓ | ✓ |
| Управлять участниками и ролями | ✗ | ✗ | ✓ |
| Архивировать проект / Design System | ✗ | ✗ | ✓ |
| Смотреть журнал действий | ✗ | ✓ | ✓ |

## 3. Главное правило: объяснять, а не прятать

Когда действие человеку недоступно, интерфейс не молчит и не даёт «мёртвую» серую кнопку без причины. Он всегда даёт понять, **почему** нельзя и **что с этим сделать**. Вот как это выглядит по типам:

| Если нельзя… | То в интерфейсе |
|---|---|
| Редактировать (поля, токены, свойства) | Поля становятся **только для просмотра**, а не просто заблокированными |
| Публиковать, создавать, откатывать | Кнопку **скрываем** (не показываем disabled); *почему* нельзя — объясняем баннером на экране, а не на каждой кнопке |
| Управлять участниками или системой (только Owner) | Раздел или действие **не показываем** тем, кто не Owner |
| Работать в теме целиком (для Viewer) | Показываем **баннер «только для просмотра»** и кнопку **«Запросить права на редактирование»** |

Это прямое следствие принципа «безопасно, но не мешая»: человек всегда понимает границы и знает, как их расширить.

## 4. Режим «только для просмотра» (для Viewer)

- Все поля — только для просмотра.
- Редактирование, публикация, создание, откат — недоступны, и понятно почему.
- Виден статус «только для просмотра».
- Есть кнопка «Запросить права на редактирование» — запрос уходит владельцам проекта.
- При этом история, версии и опубликованные конфигурации **остаются видны** — смотреть никто не запрещает.

## 5. Как ведут себя экраны для разных ролей

| Экран | Viewer | Editor | Owner |
|---|---|---|---|
| Обзор проекта | Смотрит | Смотрит | + настройки и участники |
| Design System | Смотрит | Смотрит | + создать или архивировать |
| Редактор темы | Только просмотр, баннер + запрос прав | Полная правка токенов и компонентов | Как Editor |
| Изменения | Смотрит различия | + откат своих правок | + откат |
| Публикация | Недоступна, объяснение + запрос прав | Доступна | Доступна |
| Версии | Смотрит, сравнивает | + «Восстановить как черновик» | + «Восстановить как черновик» |
| Качество | Смотрит замечания | + переход к источнику | Так же |
| Настройки проекта и участники | Нет доступа | Нет доступа | Полный доступ |

## 6. Как Owner управляет доступом

- Owner приглашает людей **только как Editor или Viewer** (сам Owner — это создатель проекта).
- Последнего Owner нельзя удалить или понизить — иначе проект останется без хозяина.
- Viewer может запросить права на редактирование; Owner выдаёт их из настроек проекта, и человек становится Editor и получает уведомление.
- Смена роли, снятие доступа, приглашение и публикация — всё идёт через одни и те же уведомления.

## 7. Что фиксируем в журнале

Создание и архивирование проектов и систем, смену ролей, создание темы, правки токенов и компонентов, проверки и публикацию. Журнал видят Editor и Owner.

## 8. Что ещё не решено

1. Насколько тонко разграничивать права на отдельные свойства компонентов — связано с общим открытым вопросом про изменяемые/закрытые свойства.
2. Нужна ли роль между Editor и Owner (например, «тот, кто только публикует») — в MVP её нет, но стоит держать как возможное расширение.
3. Права на журнал действий: сейчас его видит Editor — подтвердить с безопасностью.
