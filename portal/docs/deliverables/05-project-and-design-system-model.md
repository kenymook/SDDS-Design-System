# Модель проектов и путь от входа до работы

Спринт-задача (дизайнер команды) · [Прототип](https://portal-flow-prototype.vercel.app) → «Открыть DS Builder» (учётки `anna`, `ivan`, `maria`, пароль `demo`).

## 1. О чём задача и где границы

Нужно спроектировать всё, что человек видит **между входом и редактором темы**: своё пространство, выбор Project, Design System и Theme, переключение между ними, участников и роли проекта. Проще говоря — как человек добирается до рабочего места и не путается по дороге.

Где задача заканчивается: на входе в саму тему. Что происходит внутри редактора — это уже [границы MVP Builder](02-ds-builder-mvp-scope.md). Права ролей на экранах — [правила RBAC](03-rbac-ui-rules.md). Вход — [спецификация авторизации](01-builder-auth-spec.md).

## 2. Базовая модель

В пользовательской модели нет отдельной сущности Organization. В Builder приходят представители продуктовых команд: команда SDDS создаёт им аккаунты и выдаёт права на создание проектов и дизайн-систем.

Project — основная рабочая область и единственный уровень управления доступом. Design System и Theme находятся внутри Project и наследуют его участников и роли.

```text
User Account
  └── Project Membership (роль в конкретном Project)
        └── Project                 ← данные, доступ, участники, настройки
              └── Design System
                    ├── Components
                    └── Theme         ← вход в рабочее пространство
                          ├── Tokens / Modes / SubThemes
                          ├── Changes
                          └── Versions
```

Один человек может состоять в нескольких проектах с **разными ролями** — где-то Owner, где-то Editor, где-то Viewer.

## 3. User Account

User Account — аккаунт представителя продуктовой команды в DS Builder.

Через аккаунт пользователь получает:

- доступ к Builder;
- список доступных проектов;
- роль внутри каждого проекта;
- право создавать Project или Design System, если оно выдано командой SDDS.

## 4. Project

Project — рабочая область продуктовой команды и граница управления доступом.

В Project находятся:

- Design Systems;
- участники;
- роли доступа;
- настройки проекта;
- журнал действий;
- опубликованные версии и конфигурации.

Управление участниками происходит **только на уровне Project**. У Design System и Theme собственных ролей и участников нет.

## 5. Роли Project

| Роль | Что означает |
|---|---|
| Owner | Владелец проекта: роль с максимальной ответственностью и доступом |
| Editor | Может редактировать темы, токены, компоненты и публиковать изменения |
| Viewer | Может смотреть проект, Design System, Theme, версии и результаты без редактирования |

Owner является одновременно ответственностью за объект и ролью доступа. Создатель Project автоматически становится Owner.

## 6. Своё пространство после входа

Это не отдельная сущность, а удобный вид на все проекты человека:

- **Мои проекты** — те, где пользователь Owner;
- **Доступные мне** — куда его пригласили как Editor или Viewer.

Что происходит при входе:

- открывается последний доступный проект;
- при самом первом входе — первый доступный проект;
- если проектов нет — показываем **экран-заглушку** `builder-home`: «Создать проект» первичным действием, «Запросить доступ» вторичным.

## 7. Экраны на пути к работе

| Экран | Что на нём |
|---|---|
| Стартовый экран Builder | Панель проектов, вход в последний проект, создание проекта |
| Создать Project | Название, описание, первичная настройка, приглашения |
| Обзор Project | Design Systems проекта, участники, сводка качества, последние публикации |
| Настройки Project | Owner, участники, роли, доступы, журнал, опасная зона |
| Создать Design System | Новая дизайн-система внутри Project |
| Обзор Design System | Список тем, статус, версия, переход к настройке |
| Создать Theme | Название темы и стартовая палитра |
| Theme Editor | Рабочее пространство темы |

Отдельной страницы «все проекты» в MVP нет — переключение происходит через панель проектов.

## 8. Контекст, который всегда виден

Пока человек работает, он не должен гадать, где находится. Поэтому наверху всегда строка:

```text
Project / Design System / Theme / версия / статус / роль
пример: SberBusiness / CRM DS / Light Theme / v1.4.0 / черновик / Editor
```

## 9. Правила переключения и изоляции

- **Контекст всегда виден** — текущие Project, Design System и Theme.
- **Права пересчитываются при смене Project** — роль берётся из участия в новом проекте.
- **Design System и Theme наследуют доступы Project** — своих участников и ролей у них нет.
- **Предупреждаем о несохранённом** — если при переходе есть незакоммиченные правки, спрашиваем подтверждение.
- **Данные проектов не смешиваются** — поиск, история и публикации одного Project не пересекаются с другими.
- **Черновики у каждого свои** — пока не опубликовал, правки видит только их автор; команде достаётся опубликованное.

## 10. Сущности и их поля (для передачи разработке)

- **Project:** идентификатор, название, описание, статус (активен/архивирован), Owner, список Design System, участники, роли.
- **Project Membership:** пользователь, Project, роль (owner/editor/viewer), статус (active/invited/disabled).
- **Design System:** идентификатор, Project, название, компоненты, темы, текущая версия, ссылка на репозиторий и на библиотеку в Figma/Pixso (если есть), статус.
- **Theme:** принадлежит одной Design System; хранит токены, режимы, подтемы, изменения, версии. Значение по умолчанию унаследовано; если его переопределили — это видно и попадает в Changes.

## 11. Пустые и пограничные состояния

| Ситуация | Что показываем |
|---|---|
| Проектов нет | Пустой Builder Home + «Создать проект» / «Запросить доступ» |
| В Project нет Design System | Пустой экран + «Создать Design System» только для Owner |
| В Design System нет Theme | Пустой экран + «Создать Theme» для Owner или Editor |
| Пользователь — Viewer | Просмотр + баннер «только для просмотра» + запрос прав |
| Нет доступа к Project | No access / request access |
| Уходит с несохранёнными правками | Диалог с подтверждением |
