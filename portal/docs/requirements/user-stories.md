# User stories и acceptance criteria

> Дополнено 2026-07-06: добавлен Epic «Portal — публичный контур» (главная, DS Builder, документация, онбординг, релизы, поддержка, навигация). Ранее stories покрывали только Builder.

## Epic: Portal — публичный контур

### Знакомство с системой (главная страница)

Как потенциальный пользователь (дизайнер, разработчик или владелец продукта), я хочу за первый экран понять, что такое SDDS и зачем она моей команде.

- Первый экран: слоган, краткое описание white-label подхода и ключевые метрики (~80 компонентов; платформы Web · Mobile · TV; размерная шкала XS–XL + Scalable; WCAG 2.1 AA).
- Social proof: количество команд экосистемы и узнаваемые продукты; витрина реальных скриншотов и видео продуктов на SDDS.
- Архитектура токенов объяснена без предварительных знаний («токен — это роль, а не значение») и подкреплена визуализацией пути значения от raw до темы.
- Live demo: посетитель переключает бренд, режим (Light/Dark) и размер — меняются только значения токенов, структура демо-интерфейса неизменна.
- Блок DS Builder ведёт на отдельную страницу-презентацию инструмента; блок Developer handoff показывает CLI-команду с явной пометкой, что контракт CLI в проработке.
- Страница доступна без авторизации и адаптирована под мобильную ширину.

### Презентация DS Builder

Как посетитель портала, я хочу понять возможности DS Builder до того, как запрашивать доступ.

- Отдельная страница: путь «от черновика до Version» по шагам, проверки качества (WCAG-контраст, advisory-валидация), ролевая модель (Owner/Editor/Viewer), работа с Versions и откатом.
- CTA: открыть DS Builder (ведёт на авторизацию) и запросить доступ (ведёт в Support).
- Доступна без авторизации из навигации портала.

### Навигация портала

Как посетитель, я хочу перемещаться между разделами портала с любого устройства.

- Шапка: Главная · DS Builder · Документация · Онбординг · Релизы · Поддержка + постоянная кнопка «Открыть DS Builder»; активный раздел подсвечен.
- На узкой ширине навигация сворачивается в бургер-меню со всеми пунктами; выбор пункта закрывает меню.
- Публичные разделы (Documentation, Onboarding, Releases, Support) доступны без входа; авторизация требуется только для Builder.

### Документация

Как дизайнер или разработчик, я хочу изучать правила системы и спецификации компонентов на портале.

- Боковая навигация по страницам: foundations (Introducing, Theming, Sizes и другие) и раздел Components с группировкой по категориям.
- Страница компонента содержит живое превью и перечень свойств; из превью доступен переход «Открыть в Builder».
- Хлебные крошки отражают положение страницы (Home / Documentation / … или Components / группа / компонент).
- Доступна без авторизации.

### Онбординг по ролям

Как новый участник, я хочу пошаговый маршрут освоения системы под свою роль.

- Доступны треки Designer и Developer; трек переключается на странице.
- Шаги трека отображаются списком со статусами: выполнен, текущий, предстоящий; прогресс сохраняется отдельно для каждого трека.
- Шаг содержит инструкции и ссылки на целевые разделы (документация, Builder, релизы).
- Доступен без авторизации.

### Релизы

Как участник экосистемы, я хочу видеть историю обновлений SDDS и опубликованных конфигураций.

- Список содержит опубликованные Versions проектов (из Builder) и релизы SDDS Core с описанием изменений.
- Каждая Version связана с immutable Published Configuration для CLI.
- Открытые решения (например, CLI contract) представлены явно, а не выданы за готовое.
- Доступен без авторизации.

### Поддержка и запросы

Как пользователь портала, я хочу создать запрос: доступ, баг, новый компонент или консультацию.

- Форма: тип запроса (Запрос доступа / Консультация), команда / продукт, описание. Баги и запросы новых компонентов создаются не здесь, а в DS Builder — в контексте конкретной темы.
- При переходе из сценария сбоя (например, неудачная публикация) описание предзаполняется контекстом, а после отправки доступен возврат к исходному сценарию.
- После отправки показывается подтверждение с возможностью создать ещё запрос.
- Доступна без авторизации.

## Epic: Project context

### Просмотр и переключение Projects

Как пользователь Builder, я хочу сразу попадать в рабочий Project и быстро переключаться между доступными Projects.

- После авторизации открывается последний доступный Project; при первом входе — первый доступный.
- Общей страницы Projects нет.
- Доступные Projects отображаются иконками в узком rail; tooltip содержит название Project.
- Кнопка `+` под списком создаёт новый Project.
- Если Projects нет, сразу открывается Create Project.
- Membership определяет роль пользователя отдельно в каждом Project.

### Создание Project и Design System

Как Owner, я хочу создать Project и Design System, чтобы команда могла управлять дизайн-системой продукта.

- Создатель получает роль Owner.
- Design System принадлежит выбранному Project.
- До создания можно добавить нескольких участников и назначить каждому Editor или Viewer.
- Приглашения создают membership только в новом Project.
- Создание сущностей записывается в audit log.

### Создание Design System внутри Project

Как Owner, я хочу создать Design System в выбранном Project.

- Действие доступно только в контексте Project.
- Указывается name; platform не выбирается, потому что все системы поддерживают все платформы.
- Новая Design System принадлежит текущему Project и открывается после создания.

## Epic: Theme management

### Создание Theme

Как Editor, я хочу создать Theme на базе SDDS.

- Выбраны Project и Design System; режимы Light и Dark создаются автоматически.
- Выбрана стартовая palette; для Custom доступны пресеты и ручной ввод значений. Палитра задаёт начальные Colors и позже редактируется внутри Theme.
- Theme создаётся в Draft и наследует базовые значения.
- Создание Theme не запускает публикацию.
- Создание доступно Owner и Editor только внутри выбранной Design System.

## Epic: Tokens

### Редактирование token values

Как Editor, я хочу менять colors, sizes и fonts внутри Theme.

- Видны old/new, inherited value (Base с меткой inherited/overridden), preview и validation.
- Изменение автоматически появляется в Changes.
- Draft-изменение можно revert; работает undo/redo.
- Цвет выбирается через color picker (Custom: SV/hue/alpha, форматы Hex/RGB/HSL; Library: пресеты палитр) или вводом hex.
- Preview показывает реальные компоненты на draft-значениях; режим Accessibility считает WCAG-контраст и предлагает исправление.
- Черновики изолированы по пользователю; общие для Project — только опубликованные значения.

## Epic: Components

### Редактирование Component

Как Editor, я хочу менять разрешённые свойства компонента.

- Editable property доступно для изменения.
- Restricted property объясняет допустимые пределы.
- Locked property доступно только для просмотра.
- Изменение отображается в preview и Changes.

Конкретный перечень editable/restricted/locked properties остаётся открытым решением.

## Epic: Changes and validation

### Просмотр Changes

Как Editor, я хочу видеть полный diff перед публикацией.

- Changes сгруппированы по entity type.
- Видны old/new, author/date, issues и affected entities (считаются из модели зависимостей).
- Каждая issue сопровождается fix suggestion, применяемым в один клик.
- Review/Approve actions отсутствуют.

### Публикация с issues

Как Editor, я хочу опубликовать изменения даже при наличии validation issues.

- Warning/Error/Critical не блокируют Publish.
- Error/Critical требуют явного подтверждения риска.
- Validation snapshot сохраняется в Publication и Version.

## Epic: Publication and CLI

### Прямая публикация

Как Editor, я хочу самостоятельно опубликовать изменения без review.

- Publish доступен Editor и Owner.
- Перед подтверждением видны changes, issues, version и changelog.
- Успешная публикация создаёт Publication и Version.

### Публикация конфигурации

Как Editor, я хочу опубликовать Version, которую разработчик сможет применить через CLI.

- После Publish создаются Version и Published Configuration.
- Configuration неизменяема и связана с конкретной Version.
- При ошибке сохранения показывается Failed и Retry.
- Builder показывает инструкцию использования Version через CLI.

### Использование результата разработчиком

Как Developer, я хочу выбрать Version через CLI, чтобы сгенерировать согласованные файлы для продукта.

- CLI получает Published Configuration выбранной Version.
- Результат однозначно связан с Design System/Theme и Version.
- CLI сообщает об ошибке получения, несовместимости схемы или генерации target files.

### Сравнение и восстановление Versions

Как Editor, я хочу сравнить Version с предыдущей и вернуть её значения в работу.

- Version details показывают diff с предыдущей опубликованной Version по tokens и components.
- «Восстановить как draft» добавляет отличающиеся значения выбранной Version в Changes с подтверждением.
- Восстановленные значения проходят стандартный цикл validation → Publish; история Versions не переписывается.

### Уведомление о публикации

Как участник Project, я хочу узнавать о новых опубликованных Versions.

- После успешного Publish все участники Project, кроме публикующего, получают in-app уведомление и email.
- Уведомление содержит Project, Theme, номер Version и инициатора; открытие ведёт в Project.

## Epic: Access

### Уведомление о доступе к Project

Как приглашённый пользователь, я хочу сразу узнать, что мне дали доступ к Project.

- При приглашении система создаёт in-app уведомление и отправляет email с названием Project, ролью, инициатором и ссылкой на Project.
- Отдельное принятие приглашения не требуется: Project сразу появляется в rail.
- Непрочитанное уведомление отображается badge на колокольчике и индикатором на иконке Project.
- При первом входе показывается toast; открытие уведомления или Project помечает связанные уведомления прочитанными.
- Изменение роли, удаление доступа, запрос Editor-прав и публикация Version используют тот же канал уведомлений.

### Read-only mode

Как Viewer, я хочу просматривать дизайн-систему без возможности изменить её.

- Поля read-only.
- Edit/Publish недоступны с объяснением.
- Доступно Request edit access для обращения к Owner.
- История и опубликованные Versions остаются видимыми.

### Управление участниками

Как Owner, я хочу управлять Members и ролями.

- Доступны роли Owner, Editor и Viewer.
- Последнего Owner нельзя удалить или понизить.
- Изменения ролей записываются в audit log.

## Epic: Health

### Theme and Design System health

Как Editor, я хочу видеть issues и component coverage.

- Issues сгруппированы по severity и entity.
- Issue ведёт к источнику проблемы.
- Ошибки публикации отображаются отдельно от validation issues; ошибки генерации target files относятся к CLI.
