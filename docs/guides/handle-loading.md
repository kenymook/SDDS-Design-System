# Как обработать loading и async-операции

Async-операция в интерфейсе — это всегда три состояния: до запуска, во время выполнения, после результата. У SDDS есть отдельные паттерны для каждого.

---

## Принципы

1. **Пользователь должен видеть, что что-то происходит.** Без визуального сигнала кажется, что приложение зависло.
2. **Блокируйте повторный запуск.** Двойной клик не должен отправить два запроса.
3. **Не схлопывайте размер.** Кнопка, превратившаяся из «Сохранить» в спиннер, не должна менять ширину.
4. **Возвращайте состояние при ошибке.** Дайте пользователю возможность повторить.

---

## Кнопка с async-действием

```jsx
const [isSubmitting, setIsSubmitting] = useState(false);

const handleSubmit = async () => {
  setIsSubmitting(true);
  try {
    await api.save(data);
    showToast({ view: 'positive', text: 'Сохранено' });
  } catch (error) {
    showToast({ view: 'negative', text: 'Не удалось сохранить' });
  } finally {
    setIsSubmitting(false);
  }
};

<BasicButton
  view="accent"
  loading={isSubmitting}
  onClick={handleSubmit}
>
  Сохранить
</BasicButton>
```

Что делает `loading=true` на кнопке:

- Заменяет содержимое на спиннер
- Блокирует повторные клики (нативно)
- Фиксирует ширину — она не схлопывается
- Устанавливает `aria-busy="true"` — скринридер объявляет состояние

`loading` — **не** `disabled`. Семантически кнопка активна, операция выполняется.

## Async-валидация поля

Для проверок, требующих сервера (уникальность email, доступность логина):

```jsx
const [emailStatus, setEmailStatus] = useState('idle');

const checkEmail = async (value) => {
  if (!value) return;
  setEmailStatus('checking');
  const taken = await api.checkEmailTaken(value);
  setEmailStatus(taken ? 'taken' : 'available');
};

<TextField
  value={email}
  onChange={(e) => setEmail(e.target.value)}
  onBlur={(e) => checkEmail(e.target.value)}
  loading={emailStatus === 'checking'}
  view={
    emailStatus === 'taken' ? 'error' :
    emailStatus === 'available' ? 'success' :
    'default'
  }
  hint={emailStatus === 'taken' ? 'Email уже занят' : ''}
/>
```

Спиннер появляется в `contentRight` — пользователь видит, что идёт проверка, и не нервничает.

## Загрузка страницы или секции

Если контент ещё не пришёл — покажите Spinner на месте контента, а не где-то в углу.

```jsx
{isLoading ? (
  <Spinner aria-label="Загрузка списка заказов" />
) : (
  <OrdersList orders={orders} />
)}
```

Для долгих операций (> 5 секунд) используйте `ProgressBar` или `Loader` с прогрессом — пользователь должен понимать, сколько ещё ждать.

---

## Spinner vs Loader vs ProgressBar

| | Когда |
|---|---|
| `Spinner` | Короткое неопределённое ожидание (< 5 сек) |
| `Loader` | Загрузка с известным процентом |
| `ProgressBar` | Многошаговая операция или долгая загрузка |

Если операция затянулась — пользователь ждёт молчания и думает «зависло». Покажите прогресс хотя бы приблизительно.

---

## Что делать при ошибке

После async-ошибки:

1. **Верните элемент в исходное состояние.** Кнопка из `loading` обратно в активную.
2. **Сообщите об ошибке.** Toast `View=Negative` для системных, текст в `hint` для полевых.
3. **Дайте путь повторить.** Кнопка снова кликабельна, или есть отдельная «Повторить попытку».
4. **Верните фокус.** Если фокус был на кнопке — он остаётся на ней, пользователь может сразу повторить.

```jsx
catch (error) {
  showToast({
    view: 'negative',
    text: 'Не удалось сохранить. Попробуйте ещё раз.',
  });
  // НЕ оставляем кнопку в loading
  setIsSubmitting(false);
}
```

## Optimistic updates

Иногда полезно показать результат **до** ответа сервера:

```jsx
const handleLike = async () => {
  setLiked(true); // Сразу показываем
  try {
    await api.like();
  } catch {
    setLiked(false); // Откатываем при ошибке
    showToast({ view: 'negative', text: 'Не удалось' });
  }
};
```

Используйте для действий, которые **обычно** успешны (лайк, добавление в избранное). **Не используйте** для критичных (отправка платежа, удаление записи) — там пользователь должен видеть фактический результат.

---

## Чего избегать

| Антипаттерн | Что делает | Чем плохо |
|---|---|---|
| `loading` без блокировки кликов | Кнопка показывает спиннер, но принимает клики | Двойной клик = два запроса |
| `disabled` вместо `loading` | Кнопка просто блокируется | Пользователь не видит, что идёт операция |
| Спиннер в углу страницы при загрузке секции | Не сразу замечаешь | Кажется, что страница не работает |
| Schлопывание ширины кнопки | Контент сдвигается при появлении/исчезновении спиннера | Дёргается весь интерфейс |
| Молчание при ошибке | Кнопка вернулась, но никто не сказал, что не получилось | Пользователь думает, что всё ок |
