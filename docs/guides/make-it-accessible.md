# Как сделать интерфейс доступным

SDDS закрывает много базовых требований доступности по умолчанию: компоненты используют семантический HTML, фокус виден, пропы транслируются в ARIA-атрибуты. Но есть решения, которые остаются за вами.

Целевой уровень: **WCAG 2.1 AA**.

---

## Принципы

1. **Native first.** Если нативный HTML-элемент даёт корректное поведение — используйте его. Не пишите `<div>` с `role="button"`, если можно `<button>`.
2. **Видимый фокус.** Кольцо фокуса обязательно для всех интерактивных элементов. Не убирайте `outline` без замены.
3. **Keyboard parity.** Всё, что доступно мышью, доступно с клавиатуры.
4. **Не только цвет.** Состояние не передаётся одним только цветом — нужна иконка, текст или форма.

---

## Чек-лист для формы

### Каждое поле

- [ ] Имеет лейбл (видимый или `aria-label`)
- [ ] Если `LabelPlacement=None` — есть `aria-label` с текстом лейбла
- [ ] `Required=True` транслируется в `aria-required="true"`
- [ ] `View=Error` транслируется в `aria-invalid="true"`
- [ ] Текст ошибки связан с полем через `aria-describedby`

### Состояния

- [ ] Disabled-поля имеют атрибут `disabled` (или `aria-disabled="true"`)
- [ ] Loading-состояния помечены `aria-busy="true"`
- [ ] Открытые компоненты (Select, ComboBox) имеют `aria-expanded="true"`

### Клавиатура

- [ ] Все поля доступны через Tab
- [ ] Tab/Shift+Tab проходит поля в логическом порядке (сверху вниз)
- [ ] Submit срабатывает по Enter из любого поля
- [ ] Escape закрывает открытые dropdown'ы

---

## Чек-лист для Modal / Drawer

- [ ] При открытии фокус перемещается **внутрь** компонента (на первый интерактивный элемент или сам контейнер)
- [ ] Tab и Shift+Tab **зациклены** внутри — не выходят за пределы
- [ ] Escape закрывает компонент
- [ ] При закрытии фокус возвращается на триггер (кнопку, открывшую модалку)
- [ ] Если триггер удалён — фокус на ближайший логический элемент
- [ ] Контейнер имеет `role="dialog"` и `aria-modal="true"`
- [ ] Заголовок связан через `aria-labelledby`

## Чек-лист для уведомлений

| Компонент | `aria-live` | Когда |
|---|---|---|
| Toast `View=Negative` | `assertive` | Ошибки — объявить немедленно |
| Toast `View=Default/Positive` | `polite` | Информация — после текущего |
| Notification | `polite` | Фоновые уведомления |
| Loader / Spinner | `status` | Загрузка страницы |

`assertive` **прерывает** текущее чтение скринридером — используйте только для критичных ошибок. Для всего остального — `polite`.

---

## IconButton: всегда `aria-label`

У IconButton нет видимого текста. Без `aria-label` скринридер прочитает «button» — пользователь не узнает, что она делает.

```jsx
// ✗ Неправильно
<IconButton icon="delete" onClick={handleDelete} />

// ✓ Правильно
<IconButton icon="delete" aria-label="Удалить запись" onClick={handleDelete} />
```

То же касается `Chip` без видимого текста и кнопок закрытия.

## Контрастность

| Элемент | Минимум |
|---|---|
| Обычный текст (< 18pt) | 4.5:1 |
| Крупный текст (≥ 18pt или bold ≥ 14pt) | 3:1 |
| Иконки и UI-элементы | 3:1 |
| Disabled-состояние | Исключение WCAG 1.4.3 |

Семантические токены SDDS уже подобраны с учётом контрастности в стандартных темах. Если вы создаёте свою тему — проверьте контраст всех пар токенов в инструменте вроде [WebAIM Contrast Checker](https://webaim.org/resources/contrastchecker/).

---

## Touch-зоны

Минимальный размер touch-target: **44×44px** (WCAG 2.5.8).

Размеры XXS (20px) и XS (32px) меньше этого порога. Для них зона касания расширяется невидимым padding'ом — компонент SDDS делает это автоматически. Если вы оборачиваете компонент в свою обёртку — не схлопывайте этот padding.

---

## Цвет — не единственный сигнал

Состояние не должно передаваться только цветом. Дальтоники не различат красный и зелёный.

| Состояние | Цвет | Дополнительно |
|---|---|---|
| Error | Красный фон/обводка | Иконка ⚠ + текст ошибки |
| Success | Зелёный фон/обводка | Иконка ✓ + текст |
| Warning | Жёлтый | Иконка ⚠ + текст |

В компонентах SDDS это уже сделано. Не отключайте иконки в макетах ради «чистого» вида.

---

## Disabled vs Read-only — выбор влияет на доступность

Disabled-элемент **исключается из tab-order**. Скринридер его не озвучит. Это правильно для кнопки, которой нельзя пользоваться, но **неправильно** для значения, которое нужно показать пользователю.

| Сценарий | Используйте |
|---|---|
| Поле «Email», заполненное и нередактируемое в этом контексте | `ReadOnly=True` |
| Кнопка submit, пока форма невалидна | `Disabled=True` |
| Поле «Дата создания» в режиме просмотра | `ReadOnly=True` |
| Disabled-вариант компонента в каталоге | `Disabled=True` |

[Подробнее →](../concepts/disabled-vs-readonly.md)

## Что проверять перед релизом

1. **Tab по всей странице.** Дойдите от начала до конца только клавиатурой. Каждый интерактивный элемент должен получать видимый фокус.
2. **Скринридер на ключевых сценариях.** VoiceOver (Mac) или NVDA (Win). Послушайте, как объявляются формы, кнопки, ошибки.
3. **Контраст.** Прогоните пары цветов через checker.
4. **Без мыши.** Попробуйте выполнить основной сценарий продукта только с клавиатуры.
5. **Зум 200%.** Контент не должен ломаться при увеличении до 200%.
