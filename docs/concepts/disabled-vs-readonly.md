# `Disabled` vs `Read-only`

В SDDS два состояния «нельзя редактировать»: `Disabled` и `ReadOnly`. Они выглядят похоже, но ведут себя по-разному. Выбор между ними влияет на доступность.

---

## Различия

| | `Disabled` | `Read-only` |
|---|---|---|
| Принимает ввод | ✗ | ✗ |
| В tab-order | ✗ | ✓ |
| Значение копируется | — | ✓ |
| Зачитывается скринридером | Зависит от реализации | ✓ |
| Открывается (для Select, DatePicker) | ✗ | ✗ |
| Figma-проп | `Disabled=True` | `ReadOnly=True` |
| HTML-атрибут | `disabled` | `readonly` |

Главное отличие: **read-only остаётся в tab-order**. Пользователь может протабиться к нему, скопировать значение, услышать его в скринридере.

---

## Когда какой использовать

### Disabled — «вам сюда нельзя»

Используйте, когда поле или кнопка **временно недоступны** в принципе:

- Кнопка submit, пока форма не заполнена
- Поле даты, пока не выбрана страна (зависимое поле)
- Кнопка «Скачать», пока нет данных
- Disabled-вариант компонента в каталоге

Пользователь не должен взаимодействовать с этим элементом сейчас. Как только условие выполнится — элемент станет активным.

### Read-only — «смотрите, но не трогайте»

Используйте, когда значение **информативно**, но не подлежит редактированию в этом контексте:

- Поле «Дата создания» в форме редактирования заказа
- Поле «Email», заполненное при регистрации, нередактируемое в профиле
- Сгенерированный номер документа
- Значения в режиме просмотра (view-mode формы)

Пользователю важно **видеть** значение. Возможно, оно ему понадобится — скопировать, прочитать вслух, поделиться.

---

## Почему это важно для доступности

Disabled-элемент **исключается из tab-order**. Значит:

- Скринридер его не объявит при последовательном чтении.
- Пользователь, который ходит по форме клавиатурой, не узнает, что там вообще есть значение.
- Скопировать его значение нельзя.

Если содержимое поля **важно** — например, номер договора, который пользователь должен записать — `Disabled` его скрывает. `ReadOnly` оставляет доступ.

### Пример: форма заказа в режиме просмотра

Пользователь открыл свой завершённый заказ. Поля заполнены — редактировать нельзя. Что использовать:

- **Disabled** — все поля выключатся, скринридер ничего не прочитает, скопировать значения нельзя. ✗
- **Read-only** — все поля доступны для чтения, скринридер озвучит, можно скопировать номер заказа. ✓

Это базовый сценарий, в котором ошибка `Disabled` вместо `ReadOnly` ломает доступность.

---

## Визуальная разница

Оба состояния выглядят приглушёнными — это правильно: пользователь должен видеть, что взаимодействие невозможно.

В SDDS они различаются по силе приглушения:

- **Disabled** сильнее приглушён — текст и обводка серее. Сигнал «это сейчас неактивно».
- **Read-only** мягче — значение читаемо, но контролов нет (нет курсора-стрелочки на dropdown, нет рамки фокуса).

Конкретные токены см. в [reference/states.md](../reference/states.md).

## Что **не** Disabled

Иногда возникает соблазн отключить кнопку, потому что какие-то условия не выполнены. Часто это плохо.

| Плохой паттерн | Что вместо |
|---|---|
| Disabled submit на пустой нетронутой форме | Активный submit, валидация по клику с сообщением |
| Disabled поле с подсказкой «заполните выше» | Активное поле, валидация при попытке оставить пустым |
| Disabled во время загрузки | `Loading` (это другое состояние) |

Disabled без объяснения — фрустрирует. Пользователь видит, что нельзя нажать, но не понимает почему. Лучше дать нажать и показать сообщение, что не так.

Loading — отдельная история. Кнопка во время async-операции **не disabled**, она `loading`. Семантически это активная кнопка, выполняющая операцию.

---

## Read-only-only компоненты

Не все компоненты поддерживают `ReadOnly`. Это пропы только для:

- TextField
- Select
- TextArea

Кнопок, чекбоксов и переключателей это не касается — у них нет «значения, которое можно показать в режиме чтения». Они либо нажимаются, либо disabled.
