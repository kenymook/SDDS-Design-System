# Слоты

Все компоненты SDDS собраны из стандартных позиций — слотов. Это обеспечивает консистентную анатомию: все поля ввода устроены одинаково, все кнопки устроены одинаково, и так далее.

---

## Что такое слот

**Слот** — именованная позиция внутри компонента, в которую можно вставить контент. Позиция управляется пропом — компонент не знает, что туда попадёт, только отображает или скрывает её.

Например, у TextField есть слот `contentLeft`. Туда можно положить иконку поиска (в поле «Найти») или иконку валюты (в поле «Цена»). TextField не знает, что именно — он просто выделяет место.

```
┌────────────────────────────────────┐
│ Email *                            │  ← label
├────────────────────────────────────┤
│ [icon]  ivan@company.ru   [icon]   │  ← contentLeft, control, contentRight
├────────────────────────────────────┤
│ Введите рабочий email              │  ← hint
└────────────────────────────────────┘
```

Если `ContentLeft=False` — слот скрыт, ширина контрола подстраивается. Если `True` — слот показан, иконка размещается в нём.

---

## Зачем это нужно

### Консистентная анатомия

TextField, Select, TextArea, Autocomplete внешне разные, но устроены одинаково. У всех есть `label`, `hint`, `contentLeft`, `contentRight`, `requiredMarker`. Дизайнер один раз выучивает анатомию формы — и она работает везде.

### Композиция вместо комбинирования

Без слотов пришлось бы делать отдельные компоненты `TextFieldWithIcon`, `TextFieldWithSearchIcon`, `TextFieldWithCurrency`... Со слотами — один TextField и проп `ContentLeft=True`. Контент пихаете внутрь.

### Предсказуемость для разработки

В коде слот = именованный prop. Всегда понятно, куда что попадёт.

```jsx
<TextField
  label="Цена"
  contentLeft={<Icon name="ruble" />}
  contentRight={<Icon name="info" />}
  hint="Сумма в рублях"
/>
```

## Слоты по семействам компонентов

### Поля ввода (TextField, Select, TextArea)

| Слот | Проп | Что вмещает |
|---|---|---|
| `label` | `LabelPlacement` | Текст лейбла |
| `control` | — | Сам контрол ввода (обязателен) |
| `hint` | `HintPlacement` | Подсказка / текст ошибки |
| `contentLeft` | `ContentLeft=True` | Иконка слева внутри поля |
| `contentRight` | `ContentRight` | Иконка / спиннер / кнопка очистки справа |
| `requiredMarker` | `Required=True` + `RequiredPlacement` | Маркер `*` |

### Кнопки (BasicButton, EmbeddedButton)

| Слот | Проп | Что вмещает |
|---|---|---|
| `label` | — | Текст кнопки (обязателен) |
| `contentLeft` | `ContentLeft=True` | Иконка слева |
| `contentRight` | `ContentRight` | Иконка / счётчик / стрелка справа |
| `spinner` | `Loading=True` | Заменяет лейбл во время async-операции |

### Badge / Chip

| Слот | Проп | Что вмещает |
|---|---|---|
| `contentBefore` | `ContentBefore` | Аватар, иконка (только Chip) |
| `contentLeft` | `ContentLeft=True` | Иконка слева |
| `label` | `Label=True` | Текст |
| `contentRight` | `ContentRight=True` | Иконка / значение справа |
| `closeButton` | `hasClose=True` | Кнопка удаления (только Chip) |

## Правила использования

### Лейбл и hint — разные сообщения

Лейбл описывает поле («Email»). Hint объясняет требования или формат («Например, ivan@company.ru»). Не дублируйте смысл.

### IconButton без видимого лейбла → всегда `aria-label`

У IconButton нет слота `label`. Но скринридер должен знать, что делает кнопка. Всегда указывайте `aria-label`.

### `Required` — маркер, не текст ошибки

Маркер `*` показывает, что поле обязательно. Это **не** валидационное сообщение «поле обязательно для заполнения» — оно появляется в `hint` при `View=Error`.

### Не дублируйте иконку и текст

Если в `contentLeft` иконка поиска и в `label` написано «Поиск» — это избыточно. Оставьте что-то одно.

### Слот может быть пустым, но позиция остаётся

`ContentLeft=False` скрывает слот, ширина контрола компенсирует. Не пытайтесь «подвинуть» контент — компонент это делает сам.

---

## Расширения слотов

Иногда задача требует положить в слот что-то нестандартное — не иконку, а маленький компонент. Например, `contentRight` в TextField для async-валидации содержит спиннер. Это работает: слот принимает любой контент, который умещается в его контейнер.

Что туда **не должно** попадать:

- Длинный текст, ломающий вёрстку поля
- Интерактивные элементы, перехватывающие клики на самом поле (если это не предусмотрено компонентом)
- Контент, конфликтующий с состояниями (например, своя иконка ошибки при `View=Error` — она уже отрисована системой)
