# Зачем семантические токены

Самый частый вопрос новичков в SDDS: «зачем такие длинные имена токенов? Почему не просто `red` или `primary-color`?»

Короткий ответ: семантические токены позволяют **менять тему без правки компонентов**. Длинный — ниже.

---

## Сравнение трёх подходов

### Подход 1: хардкод

```css
.button-danger {
  background: #D32F2F;
  color: white;
}
```

Просто. Работает. Сломается при первой же попытке сделать тёмную тему — нужно переписывать каждое правило. Несколько брендов? Скопировать весь CSS и поменять цвета. Любая правка палитры — поиск всех `#D32F2F` по проекту.

### Подход 2: переменные с цветовыми именами

```css
:root {
  --red: #D32F2F;
  --green: #4CAF50;
}

.button-danger {
  background: var(--red);
}
```

Лучше — есть одна точка изменения цвета. Но имена врут. В тёмной теме «красный для ошибки» может быть `#FF6B6B` — а переменная всё ещё называется `--red`. Если бренд хочет, чтобы ошибки были оранжевыми — переменная `--red` хранит оранжевый. Имя теряет смысл.

### Подход 3: семантические токены

```css
:root {
  --surfaces-default-status-solid-negative: #D32F2F;
}

.button-danger {
  background: var(--surfaces-default-status-solid-negative);
}
```

Имя описывает **роль**, не значение. «Поверхность для негативного смысла». В тёмной теме это `#FF6B6B`, в оранжевом бренде — оранжевый, но имя не меняется. Компонент не знает, какой там цвет — он знает только смысл.

---

## Что это даёт

### Один компонент — много тем

```css
/* light theme */
:root {
  --text-icons-default-general-primary: #1A1A1A;
  --surfaces-default-general-solid-card: #FFFFFF;
}

/* dark theme */
:root[data-theme="dark"] {
  --text-icons-default-general-primary: #FFFFFF;
  --surfaces-default-general-solid-card: #2A2A2A;
}
```

Компонент использует токены — он автоматически работает в обеих темах:

```css
.card {
  background: var(--surfaces-default-general-solid-card);
  color: var(--text-icons-default-general-primary);
}
```

Никакого `@media (prefers-color-scheme: dark)` внутри компонента. Никакого дублирования стилей.

### Бренды без правки компонентов

Sber Online использует синий акцент. Sber Business — зелёный. Один и тот же компонент `BasicButton View=Accent` корректно отображается в обоих — потому что:

```
В коде компонента:
  background: var(--surfaces-default-accent-solid-accent);

В теме Sber Online:
  --surfaces-default-accent-solid-accent: #2196F3;

В теме Sber Business:
  --surfaces-default-accent-solid-accent: #4CAF50;
```

Компонент идентичен. Тема разная.

### Имена остаются осмысленными при смене значений

Через год дизайн-команда решит, что цвет ошибки слишком яркий, и сменит его с `#D32F2F` на `#B71C1C`. Имя `--surfaces-default-status-solid-negative` остаётся — это всё ещё «поверхность для ошибки», просто значение другое. Никто не переименовывает переменные в проекте.

## Структура имени

```
Surfaces / Default / Status / Solid / Negative
   ↑          ↑         ↑        ↑       ↑
категория  контекст   группа   тип    вариант
```

| Сегмент | Что описывает | Примеры |
|---|---|---|
| **Категория** | Что это: фон, текст, обводка, наложение | `Surfaces`, `Text&Icons`, `Outlines`, `BG`, `Overlay` |
| **Контекст** | На какой поверхности находится | `Default`, `Inverse`, `onDark`, `onLight` |
| **Группа** | Семантическое назначение | `General`, `Accent`, `Status` |
| **Тип** | Прозрачный или непрозрачный | `Solid`, `Transparent` |
| **Вариант** | Конкретная роль внутри группы | `Primary`, `Negative`, `Accent`, `AccentMinor` |

Имя длинное специально. Это не недостаток — это документация, встроенная в код.

---

## Контексты — отдельная важная штука

Контекст описывает **поверхность, на которой находится компонент**:

| Контекст | Где |
|---|---|
| `Default` | На основном фоне страницы (зависит от темы) |
| `Inverse` | На инвертированном фоне (зависит от темы) |
| `onDark` | Гарантированно тёмный фон (не зависит от темы) |
| `onLight` | Гарантированно светлый фон (не зависит от темы) |

Зачем это нужно: тёмная плашка с акцентным CTA должна оставаться читаемой и в светлой, и в тёмной теме. Используете токен с контекстом `onDark` — текст всегда контрастный к тёмному фону.

---

## Что считается «нарушением»

В коде компонента SDDS **не должно быть**:

```css
/* ✗ Хардкод */
background: #FF0000;

/* ✗ Цветовое имя без семантики */
background: var(--red);

/* ✗ Полупрозрачный hex поверх токена */
background: rgba(0, 0, 0, 0.5); /* надо использовать --overlay-default-soft */
```

Должно быть:

```css
/* ✓ Семантический токен */
background: var(--surfaces-default-status-solid-negative);
```

Если для какого-то случая нет подходящего токена — это значит, в системе нужен новый токен, а не хардкод в компоненте.
